### Bug: Pre-dinner gap showing as idle time / single activity stretched to implausible duration

**Root cause:** `stretchPreDinnerGap` in `api/generate-resolved-itinerary.js` was absorbing the entire pre-dinner gap into a single activity (no cap), producing e.g. a 300-minute museum visit.

**Fix:** Rewrote `stretchPreDinnerGap` to distribute the gap backwards across all afternoon activities, each capped at `MAX_SINGLE_ACTIVITY_DURATION_MINUTES = 150`. Works backwards from the last pre-dinner activity: if it would need to exceed 150 min to absorb the whole gap, it is capped at 150 and the remainder spills to the activity before it, and so on. Once durations are set, dinner's `startTime` is written directly (bypassing `realignScheduleTimes`' drift-tolerance guard) so the schedule stays fully consistent.

Key constants:
```js
const MIN_GAP_TO_STRETCH_MINUTES = 30;
const MAX_SINGLE_ACTIVITY_DURATION_MINUTES = 150;
const PRE_DINNER_BUFFER_MINUTES = 15;
```

### Bug: Days where AI omits dinner entirely (day ending at 17:04 with no dinner)

**Root cause:** Claude occasionally generates a day with no dinner item at all.

**Fix:** Added `ensureDinner(day, destination)` in `api/generate-resolved-itinerary.js`. Runs after `enforceMealConstraints` and before `applyAccommodationBookends`. If no dinner item exists, injects a placeholder at 19:30 with a generic description telling the traveller to find a local restaurant. This means no specific restaurant name — the description signals it's a suggestion, not a booking.

Wired into the processing loop:
```js
itinerary.days.forEach((day) => {
  enforceMealConstraints(day);
  ensureDinner(day, destination);
  applyAccommodationBookends(day, accommodationDetails);
  enforceEarliestStart(day);
});
```

### Feature: Geographic routing logic + 60-minute travel time hard cap

**Prompt-side (`api/_lib/generateRawItinerary.js`):** Added CRITICAL ROUTING RULE to both the packed and slow variant prompts:

> All stops every day must be within the destination city or its immediate urban area — no day trips to towns or attractions in other cities, even if they are famous. Group stops by neighbourhood or district so the traveller moves in one direction across the city rather than darting back and forth. Morning stops should cluster together, afternoon stops should cluster together near the lunch spot, and dinner should be near the last afternoon activity or en route back to the hotel. Travel time between any two consecutive stops must never exceed 60 minutes by any mode of transport — if a place would take longer than 60 minutes to reach from the previous stop, do not include it; choose something closer instead.

**Server-side (`api/generate-resolved-itinerary.js`):**
- `MAX_SAME_DAY_TRAVEL_MINUTES` reduced from 120 to 60.
- `enforceDriveCap` mode restriction removed (previously only caught drive legs; now catches walks too).
- `enforceDriveCap` gained a category fallback search: if no same-named nearby replacement exists (because the named place is in a different city entirely), it falls back to searching by `next.type` alone near the current stop's location, so a replacement is still found rather than leaving the out-of-range stop in place.

### Bug: Pending recompute loading state missing after swap/reorder

**Root cause:** After a swap or reorder, real travel times were being refetched silently with no on-screen indication. On slow connections this looked broken — travel indicators disappeared for up to a minute with no feedback.

**Fix (two parts):**
1. `TripContext.jsx` gained `pendingRecomputes` (a `Set` keyed by `"variantKey:dayIndex"`) and `recomputeRequestIds` (a `useRef` map) to track in-flight recomputes per day. `isDayRecomputing(variantKey, dayIndex)` lets screens show a real loading state. A monotonically increasing request ID per day guards against stale responses overwriting newer data when two swaps land on the same day before the first request returns.
2. `fillMissingTravelTimes` in `api/_lib/scheduleRealign.js` parallelised: was a sequential `for` loop, now a `Promise.all` — each gap's estimate is independent, so there was no reason to stack them.

### Bug: Jumbled itinerary — time goes backwards, duplicate breakfasts, no lunch

**Root cause:** Claude occasionally returns a day's `items` array in non-chronological order (e.g. a breakfast item with `startTime: "09:00"` at array index 3, after items timestamped 11:00 and 13:00). Every downstream step — `allItems` flat-list indexing, `realignScheduleTimes`' `i-1→i` chain, `stretchPreDinnerGap`'s "last pre-dinner activity" scan — assumes items are in time order. An out-of-order array produces a visibly broken schedule.

**Investigation note:** This was initially suspected to be cross-variant data contamination (items from Slow appearing inside Packed). After reading every file in the pipeline (`TripContext.jsx`, `ComparisonView.jsx`, `verifyPlace.js`, `refreshDescriptions.js`, `scheduleRealign.js`, `travelTime.js`, `estimateTravelDuration.js`, `generateRawItinerary.js`, `generate-resolved-itinerary.js`), there is no cross-variant contamination in the code — both `resolveItinerary` calls are fully independent (separate `JSON.parse` outputs, separate local `allItems` arrays, separate `usedPlaceIds` sets, no module-level shared mutable state anywhere). The real cause is ordering.

**Fix:** Sort-by-`startTime` added at the very top of `resolveItinerary`, before anything else touches the array:

```js
itinerary.days.forEach((day) => {
  day.items.sort((a, b) => {
    const aMin = timeToMinutes(a.startTime);
    const bMin = timeToMinutes(b.startTime);
    if (aMin == null && bMin == null) return 0;
    if (aMin == null) return 1;
    if (bMin == null) return -1;
    return aMin - bMin;
  });
});
```

### Bug: Phantom gap before lunch — schedule shows impossible times

**Root cause:** `stretchPreDinnerGap` collected all non-meal activities before dinner, including morning ones. If a morning activity (e.g. a 3-hour ski session) was extended past lunchtime, `realignScheduleTimes` would push lunch's candidate time forward to the correct cascaded value — but if that drift exceeded `MEAL_DRIFT_TOLERANCE_MINUTES = 60`, the guard snapped lunch back to its original AI-planned time. Result: the card showed the traveller at lunch before they had physically left the morning stop. A genuine impossible schedule with no visible explanation.

**Fix:** `stretchPreDinnerGap` now only collects activities that fall after lunch, not all activities before dinner:

```js
const lunchIndex = day.items.findIndex((item) => item.mealType === 'lunch');
const afternoonStart = lunchIndex >= 0 ? lunchIndex + 1 : 0;
const afternoonActivities = [];
for (let i = afternoonStart; i < dinnerIndex; i++) {
  // collect only post-lunch, pre-dinner activities
}
```

## Established conventions (additions to previous handoffs)

- **Git push must be done from the user's terminal**, not the device bridge shell — `git push` from within a mounted path fails with a proxy error. The typical flow is: Claude writes the file via `device_commit_files`, then asks the user to run `git add / git commit / git push` in their terminal. Lock file errors (`HEAD.lock`) are fixed with `rm -f .git/HEAD.lock` before re-running.
- **`stretchPreDinnerGap` must only touch post-lunch activities** — extending morning activities past lunchtime creates an impossible schedule that the `MEAL_DRIFT_TOLERANCE_MINUTES` guard then permanently locks in. The post-lunch restriction was introduced this session and must not be reverted.
- **Cross-variant contamination is not possible in the current architecture** — both `resolveItinerary` calls are fully independent. If a jumbled itinerary is ever reported again, the cause is ordering (Claude's output), not data mixing. Look at the raw Claude response first.
- **`MEAL_DRIFT_TOLERANCE_MINUTES = 60`** (in `scheduleRealign.js`) is a deliberately wide window. It becomes a problem only when an activity's duration is artificially extended past a meal's window — that's the root-cause pattern, not the tolerance value itself. Don't tighten the tolerance; fix whatever is pushing the activity past the meal.
